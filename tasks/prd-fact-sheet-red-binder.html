Overview

Replace the current flat label/value fact sheet with a structured "red binder" for each property: categorized Contacts (key people, vendors, emergency/utility) plus a secure Credentials vault (vendor logins, fire-alarm code, lock boxes). A standardized template applies to all ~31 properties so a cold GM or traveling VP can walk into any hotel and run it. The sheet auto-formats the property page into clean, scannable sections and retires the physical binder + post-it notes — the "State of the Union" Chris called the "lifeblood of a hotel site."

📂Source data: Who to Call List.xlsx + Password list blank.xlsx (the structure drives the template; values are entered in-app, not imported).

Goals

  • Structured Contacts — name, phone, mobile, email, account #, fax — grouped into People, Vendors, and Emergency & Utilities.
  • Secure Credentials vault — system/vendor, account #, username, password. Passwords encrypted at rest, shown masked with click-to-reveal, and every reveal audited.
  • Standardized template across every property (predefined rows from the two lists); GMs fill values (start blank); custom rows allowed.
  • Visibility — a GM sees their own property's full sheet; admin/Strand see all. Enforced at page, list-API, and reveal-API layers.
  • Auto-formatted property page — sectioned, responsive, print-friendly; replaces the flat fact sheet.

Non-Goals

  • Not importing the real Excel values — template ships blank.
  • Not a general password manager — no autofill, no external sharing.
  • No per-section custom visibility toggles in v1 (fixed GM-own / corporate-all).
  • No secret rotation, expiry, or 2FA in v1.
  • No changes to the document queue, financials, or property files.

Assumptions

  • Who fills it: GMs maintain their own property's values; admin/Strand can edit any property.
  • Encryption key lives in a Cloudflare Worker secret (CREDENTIALS_KEY), never in source.
  • Existing flat facts are preserved as a "Notes / Other" section (light migration), not discarded.

User Stories · each one session

1

Data model + migration

Create property_contacts (category, label, contact_name, phone, mobile, email, account_number, fax, sort_order, is_custom) and property_credentials (label, account_number, username, password_ciphertext, password_iv, sort_order, is_custom). Indexes on property_id.

Acceptance
Migration applies cleanly to remote D1; both tables exist with the columns above; no impact on existing tables; schema.sql updated to match.
2

Standardized template definition + apply

Canonical template in lib/factsheet-template.ts from the two lists (People, Vendors, Emergency/Utility, Credentials). Apply blank rows to every active property; auto-apply on new-property creation.

Acceptance
Every active property gets the full template set with empty values; new properties auto-seed; re-running is idempotent (no duplicates).
3

Contacts — view + edit on the property page

Render the three contact categories with typed fields; inline edit/save for authorized users; add/remove custom rows per category.

Acceptance
Contacts render grouped; edit saves and persists; a GM can edit only their own property; empty fields show "—"; custom rows add/remove.
4

Credentials vault — store + mask + reveal (audited)

Passwords masked by default. A Reveal control calls POST /api/credentials/:id/reveal → authorize → decrypt server-side → return value → write audit event. Create/edit encrypts (AES-GCM) before storing.

Acceptance
Passwords never appear in the initial HTML; Reveal shows the value and writes an audit_events row (credential id + property); stored values are ciphertext in D1; editing re-encrypts.
5

Access control + verification

Enforce GM-own / corporate-all at the page guard, the list queries, and the reveal endpoint (mirrors the documents-download pattern).

Acceptance
A GM cannot load or reveal another property's contacts/credentials (even with a guessed id); admin/Strand can; every reveal is audited.
6

Property-page formatting + polish

Lay out as sections (People / Vendors / Emergency & Utilities / Credentials / Notes), responsive and print-friendly, replacing the flat fact sheet on the GM and admin property pages.

Acceptance
Property page shows the five sections in order; mobile-readable; "Print fact sheet" renders the binder cleanly (credentials masked unless revealed); existing facts appear under Notes.

Technical Considerations

  • Encryption (D1 has no column encryption): AES-GCM via WebCrypto in the Worker; store ciphertext + iv; key from Worker secret CREDENTIALS_KEY; decrypt only in the reveal endpoint, server-side, over HTTPS.
  • Reveal endpoint POST /api/credentials/:id/reveal → auth + ownership → decrypt → return {value} → audit. Never ship plaintext in page loads.
  • Audit — reuse audit_events; add credential.reveal, credential.update, contact.update.
  • Visibility — reuse middleware gates; per-row ownership checks like api/documents/[id]/download.ts (GM query always AND property_id = ?).
  • Template — canonical list in code; admin "apply template" action + auto-apply on property creation; idempotent.
  • Migration of existing facts — keep property_facts; surface under "Notes / Other."
  • Reuse — follow existing fact/API/component patterns; split into ContactsSheet, CredentialsVault, Notes.

Open questions

  • Single Worker secret for the AES key is fine for one client — confirm.
  • "Print fact sheet": keep credentials masked (default) or allow a full authorized reveal for printing?
