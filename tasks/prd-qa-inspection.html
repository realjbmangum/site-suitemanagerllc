Overview

Replace the ASA Quality Assurance spreadsheet with a mobile-first inspection app inside the portal. An ops leader walks a property with a phone and only taps deficiencies; the app computes a score, writes it to the property (feeding history/trends), generates a deficiency punch-list, and creates follow-up work orders. It works offline because the walk hits stairwells and back-of-house dead zones. The QA score gates GM bonus eligibility (90+ eligible, 80+ "safe zone"), so accuracy, history, and an auditable record matter.

📋Source: ASA Quality Assurance (1).xlsx — Exterior & Admin checklist (Curb Appeal, Laundry/Gym, Courtyard, Storage/Housekeeping, Office-Lobby, Manager & Staff, Maintenance) + per-room inspections (Living Room / Kitchen / Bath / Bedroom).

Goals

  • Fast mobile "walk & tap" — assume-pass; tap only fails/N-A; big targets, one section at a time, minimal typing.
  • Digitize the full checklist — every Exterior & Admin section + per-room item across a room sample.
  • Scoring — applicable passed ÷ applicable (N/A excluded) × 100; per-section subtotals + overall; snapshotted at submit.
  • Offline-capable PWA — full inspection with no signal; autosave locally; sync (submit + photos) when back online.
  • Outputs — score→property (history/trends/rollups), auto punch-list, assignable work orders, photos on fails.
  • Data-driven, versioned template — admin-editable checklist; past inspections keep their version + score.

Non-Goals (v1)

  • Native iOS/Android apps (PWA only).
  • Per-item weighting (equal weight v1; schema reserves weight).
  • Real-time multi-inspector collaboration on one inspection.
  • Auto-scheduling QA visits (separate effort — CRM #223).
  • GMs self-inspecting (ops/admin run it; GM gets a read-only result).

Build plan · 15 stories / 5 phases

1 Data model & template foundation

S1Schema + migration

qa_templates / template_sections / template_items (scope: property | room; weight default 1) · qa_inspections (template_version, status, score snapshot) · inspection_rooms · qa_responses (result pass|fail|na, comment, photo) · qa_followups.

Done when migration applies to D1; all tables present; schema.sql updated.

S2Seed template from the spreadsheet

Parse the xlsx into sections + items (Exterior & Admin + room sub-areas), scoped property vs room, as template v1.

Done when the full checklist exists as template rows; one active version.
2 Inspection flow (mobile, online first) usable inspector

S3Start an inspection

Pick a property (or deep-link from its page), confirm rooms to inspect, create an in-progress inspection (client-generated id for offline).

Done when a new inspection + rooms are created; resumes an existing in-progress one.

S4Exterior & Admin checklist UI

Section-by-section; each item defaults to pass; tap → fail (reveals comment/photo) or N/A; sticky progress + running subtotal; large tap targets; autosave per change.

Done when default-pass works with no taps; progress/subtotal live; each change saves.

S5Per-room inspection UI

Room item-set per sampled room (Living Room/Kitchen/Bath/Bedroom); swipe/next between rooms; per-room completion.

Done when each room has its own responses; movement + progress work.

S6Scoring + submit

Equal-weight score (applicable = pass+fail) per section + overall; submit snapshots score and writes the property's current score.

Done when score matches a hand calc; N/A excluded; submit locks the inspection + updates the property.
3 Outputs the payoff

S7Photos on fails

Capture/select a photo on a fail; compress client-side; upload to R2 (qa/{inspectionId}/{responseId}); thumbnail in the punch-list.

Done when a fail can carry a photo that uploads (or queues offline) + renders.

S8Punch-list + PDF

Compile all fails (item, section/room, comment, photo) into a clean list; export/save a PDF to the property's records.

Done when the list = exactly the fails; PDF saves to records + downloads.

S9Follow-up work orders

Turn fails into assignable qa_followups (open→done, due date); list + resolve; open-count on the property.

Done when fails become tracked tasks; assignee/status persist; resolving updates the count.

S10Score history, trends & rollups

QA dashboard: per-property trend, best/worst, group/region averages; latest score on the property page + fact-sheet header.

Done when dashboard shows trends/rollups; property shows latest score + date.
4 Offline PWA the hard part

S11PWA shell + service worker

Installable; cache app shell + active template + selected property for offline use.

Done when the inspection route loads with no network after first visit; installable on a phone.

S12Offline inspection state

Responses + photos in IndexedDB; autosave; resume after closing; fully offline.

Done when a full inspection (incl. photos) works in airplane mode + persists across restarts.

S13Sync engine

On reconnect, submit queued inspections + upload queued photos; idempotent (client ids); show sync status/failures.

Done when offline inspections sync cleanly; re-sync makes no dupes; failures are visible + retryable.
5 Admin & access harden

S14Template management

Admin adds/edits/reorders/deactivates items, bumping the template version; past inspections keep theirs.

Done when the checklist changes without code; new inspections use the new version; old scores unchanged.

S15Roles + GM read view

Inspecting limited to admin/strand; GM sees their property's latest score + punch-list read-only; enforced at page + API.

Done when a GM can't start/edit/see other properties; can view their own result.

Technical Considerations

  • Module, not standalone — in app-suitemanager (Astro + CF Pages + D1 + R2); mobile-first routes (/qa); reuses session auth + properties + canAccessProperty.
  • Offline is the hard part — service worker caches shell + template + property; IndexedDB holds in-progress state + photo blobs; sync flushes on reconnect. Client-generated ids → idempotent upserts. Photos compressed client-side before R2 upload.
  • Scoring snapshot + versioning — store template_version on each inspection; snapshot score/pass/applicable at submit so template edits never rewrite history.
  • Score surfacing — property's current score = latest submitted; show on Overview + fact-sheet header; trends/rollups in a QA dashboard (could later feed GM bonus logic).
  • Reuse infra — PDF punch-list saved as a property record (R2); follow-ups a new lightweight table on the property page.
  • Sequence it — Phases 1→3 = a usable inspector + outputs (ship + get Mike/Mel using it); Phase 4 adds offline; Phase 5 hardens.
